Proxy Browser Usage Guide

Public install links
- Android PWA: https://robertzhangwei1.github.io/proxy-pwa-launcher/android.html
- iOS PWA: https://robertzhangwei1.github.io/proxy-pwa-launcher/ios.html

What this app does
- This app is a remote browser shell for a hosted Chromium session running on Fly.io.
- The hosted Chromium session is started with the built-in iProyal proxy configuration.
- The browser traffic from that hosted Chromium session goes through the proxy.
- The phone app shows a live remote page surface and sends navigation, tap, scroll, and typing actions to that hosted browser.

What is routed through the proxy
- The website loaded by the hosted Chromium session.
- All page requests made by that hosted Chromium session, including HTML, CSS, JavaScript, images, and in-page API requests.
- Any site you enter in the app URL bar after the browser is launched.

What is NOT routed through the proxy
- The Safari, Chrome, or Edge tab that displays the app shell itself.
- The phone's normal browsing outside this app.
- GitHub Pages assets for the app interface.

Important limitation
- This is a fully interactive remote browser, not a native local Safari or Chrome tab.
- The real browser engine is running on Fly.io.
- The phone app is controlling that remote browser and showing its live page image.

How to use it
1. Open the Android or iPhone install link on the phone.
2. Install the page as a PWA.
3. Open the installed app.
4. Tap Launch Browser.
5. Use the URL bar to open any website you want.
6. Tap the live page surface to click links or focus fields.
7. Swipe up or down on the live page surface to scroll remotely.
8. Type in the Typing Tray and tap Send Text to send text to the focused field.
9. Use Back, Forward, Reload, and Stop just like a browser toolbar.

How the routing works step by step
1. The app sends a request to https://proxy-pwa-launcher-robertzhangwei1.fly.dev.
2. The Fly backend launches Chromium with the built-in iProyal proxy configuration.
3. Chromium opens the target site in a mobile viewport.
4. The app polls session status and screenshots from the backend.
5. When you tap, scroll, type, go back, go forward, reload, or navigate, the app sends those actions to the hosted browser.
6. The hosted browser performs those actions and keeps routing page traffic through the proxy.

Per-user behavior
- Each install keeps its own private session token in local storage.
- Different users get isolated browser sessions.
- One user does not replace another user's page, cookies, screenshot stream, or browsing history.

Current default behavior
- Launch Browser starts the hosted browser immediately.
- If the URL bar is empty, the app opens the default target page.
- The current default target page is https://www.google.com/.
- Proxy host, port, username, and password are already built into the backend, so users do not need to enter them.

What to expect
- Page updates are near-live screenshot refreshes, not a true local WebView tab.
- Some highly dynamic sites may feel a little slower than a native browser because every interaction goes through the remote backend.
- The browser session stays isolated to your app install token until you tap Stop or the backend times it out.
